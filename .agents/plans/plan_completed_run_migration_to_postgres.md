# Implementation Plan: Migrate Completed Runs from Local Disk to PostgreSQL

## Goal

Migrate completed workflow run persistence from local JSON files to PostgreSQL-backed completed run archival.

The purpose of completed run persistence is:

audit
history
observability
workflow analytics
CLI run inspection
future RAG document generation
backtest/run comparison

Completed runs are not resumable execution state.

Resume/replay remains owned by:

CheckpointManager
ReplayEngine
Core Architecture Decision
Correct Separation
CheckpointManager
    = resumable execution state
    = replay/recovery
    = execution position metadata
    = checkpoint lifecycle

CompletedRunArchive
    = completed workflow history
    = final context snapshot
    = audit/archive
    = queryable history
    = source material for curated RAG records

Do not use completed runs for workflow resume.

Do not use checkpoints as the completed run archive.

# New Codex Proposed Implementation Plan

## Summary

Completed workflow run archival will be migrated fully to PostgreSQL.

Key decisions:

- PostgreSQL becomes the canonical completed-run archive backend.
- `LocalDiskCompletedRunArchive` will be removed.
- Completed runs remain historical/audit records, not resumable checkpoints.
- Checkpointing and replay remain owned by `CheckpointManager` and `ReplayEngine`.
- The completed-run archive contract will become async to match the platform's async PostgreSQL persistence architecture.
- Existing local JSON completed-run files will not be migrated automatically in this pass unless a separate import tool is requested later.
