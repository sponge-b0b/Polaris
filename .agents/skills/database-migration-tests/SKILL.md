# Database Migration Tests

## Objective
Ensure every schema modification preserves data integrity, avoids destructive edits to canonical tables, and cleanly rolls back without orphaned database states.

## Initial Pre-flight Check
Before writing or executing any database schema modification:
1. Identify the authoritative SQLAlchemy models being altered.
2. Confirm that changes are driven purely via `alembic` migration scripts.
3. **Constraint:** Projection rebuilds (e.g., Qdrant, Neo4j) are strictly forbidden from deleting canonical PostgreSQL source records.

## Execution Steps

### Step 1: Structural Change Mapping
Draft or inspect the target migration. Do not promote arbitrary metadata into a durable schema layout; ensure new concepts are implemented as first-class, strictly typed columns.

### Step 2: Generate the Revision Script
Run the automated generation tool via `uv` or your environment's package proxy:
```bash
uv run alembic revision --autogenerate -m "type: descriptive migration intent"
```

### Step 3: Audit the Autogenerated Script
Open the newly generated migration file and verify:
- Explicit column types match your frozen dataclass structures.
- Foreign key restraints specify clear cascade actions.
- Indexes are applied cleanly to high-churn lookups.

### Step 4: Run the UP / DOWN Lifecycle Validation
You must execute a full round-trip execution matrix to guarantee the migration script is fully reversible:
1. **Apply Upgrade:** Run the upgrade command to test your `upgrade()` block.
   ```bash
   uv run alembic upgrade head
   ```
2. **Verify State:** Confirm tables and constraints match expected shapes.
3. **Apply Downgrade:** Revert the change immediately by exactly one step to test your `downgrade()` block.
   ```bash
   uv run alembic downgrade -1
   ```
4. **Final Verification:** Confirm your local database schema returns precisely to its baseline state with zero lingering structures or locks.
